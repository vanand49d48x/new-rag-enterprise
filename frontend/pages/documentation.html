Overview

The Enterprise RAG System is a production-ready, multi-modal Retrieval-Augmented Generation platform designed for enterprise deployments. It combines advanced document processing, intelligent search, and AI-powered question answering with enterprise-grade reliability and monitoring.

What is RAG?

Retrieval-Augmented Generation (RAG) combines document search with AI generation to provide accurate, source-cited responses based on your specific documents and data.

Key Benefits

Multi-Modal Processing

Support for PDF, DOCX, TXT, CSV, images, audio, and video files with OCR and transcription capabilities.

Intelligent Search

Hybrid search combining semantic, BM25, and cross-encoder re-ranking for optimal results.

Enterprise Monitoring

Comprehensive monitoring with Prometheus and Grafana dashboards for production deployments.

Auto-Configuration

Intelligent system detection and automatic configuration optimization for your hardware.

Quick Start

Get up and running with the Enterprise RAG System in minutes.

Prerequisites

  • Docker and Docker Compose installed
  • 4GB+ RAM available
  • 10GB+ storage space
  • Internet connection for model download

One-Command Setup

# Clone and start the system git clone <repository-url> cd rag-enterprise ./start.sh

What happens automatically:

  • System architecture detection (ARM64/AMD64)
  • Optimal model download and configuration
  • Docker services startup and health checks
  • Web interface availability

Configuration

Auto-Configuration System

The system automatically detects your hardware and generates optimal configurations:

# Generate optimal configuration for your system ./scripts/generate_config.sh # Force specific tier (laptop/workstation/server/enterprise) ./scripts/generate_config.sh --tier laptop # Apply customer overrides cp configs/auto_generated.env configs/customer_overrides.env # Edit the overrides file, then regenerate ./scripts/generate_config.sh --force

Performance Parameters

Key Configuration Parameters

  • Context Size: Maximum tokens for processing (1024-4096)
  • Batch Size: Tokens processed in parallel (64-512)
  • Threads: CPU threads for processing (2-16)
  • GPU Layers: GPU acceleration layers (0-35)

Troubleshooting

Common Issues

Model Download Fails

Solution: Check internet connection and try manual download:

# Check connection curl -I https://huggingface.co # Manual download wget https://huggingface.co/Qwen/Qwen2-7B-Instruct-GGUF/resolve/main/qwen2-7b-instruct-q4_k_m.gguf

LLM Timeout

Solution: Use smaller model or increase timeout:

# Use smaller model MODEL_CONFIG=tinyllama ./start.sh # Or increase timeout in config timeout: 300 # 5 minutes

Memory Issues

Solution: Reduce model threads or use smaller model:

# Check available memory free -h # Reduce threads in config threads: 4 # Instead of 8

Health Checks

# Check service status docker-compose ps # View logs docker-compose logs -f backend docker-compose logs -f llama-cpp # Health endpoints curl http://localhost:8000/ # API health curl http://localhost:6334/collections # Qdrant health curl http://localhost:8080/health # LLM health
